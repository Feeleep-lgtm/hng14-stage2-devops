# FIXES

| File | Original line(s) | Problem | Change made |
| --- | --- | --- | --- |
| `api/main.py` | 8 | Redis was hardcoded to `localhost:6379`, which works on a developer machine but breaks when the API runs in its own container because `localhost` points to the API container, not the Redis service. | Replaced the hardcoded Redis connection with environment-driven `REDIS_HOST`, `REDIS_PORT`, and `REDIS_QUEUE_NAME` settings, with safe fallbacks for local development. |
| `api/main.py` | 13-14 | The API pushed the job ID onto Redis before writing the job status. A fast worker could pop and finish the job before the `queued` status was stored, causing the API to overwrite a completed job back to `queued`. | Wrote the job status first and only then pushed the job ID onto the queue so the worker can never beat the initial state write. |
| `api/main.py` | 19-21 | Missing jobs returned HTTP 200 with a JSON error payload, which makes failed lookups appear successful to clients and upstream services. | Changed the endpoint to raise `HTTPException(status_code=404)` for missing jobs and kept successful responses in the normal success shape. |
| `api/main.py` | 19, 22 | Redis returned byte strings, so the handler had to decode manually and mixed raw Redis response handling into the endpoint logic. | Enabled `decode_responses=True` on the Redis client and returned plain strings directly from the handler. |
| `api/main.py` | 9-13 | The API did not support authenticated Redis connections, so enabling a Redis password in containers would immediately break job creation and status reads. | Added optional `REDIS_PASSWORD` support to the Redis client and exposed a `/health` endpoint that verifies Redis connectivity. |
| `worker/worker.py` | 6 | The worker was also hardcoded to `localhost:6379`, so it would fail to reach Redis when deployed as a separate container. | Switched the worker to the same environment-driven Redis host, port, and queue settings used by the API. |
| `worker/worker.py` | 11 | Jobs moved directly from `queued` to `completed`, so the UI had no visible in-progress state while the worker was sleeping and processing. | Added a `processing` status update before the simulated work starts. |
| `worker/worker.py` | 15 | The worker blocked on a hardcoded queue name, which would desynchronize it from the API if the queue name changed by environment. | Replaced the literal queue name with the shared `REDIS_QUEUE_NAME` configuration. |
| `worker/worker.py` | 15-18 | Any transient Redis failure during `BRPOP` would crash the worker loop instead of retrying, which is especially brittle during container startup ordering. | Wrapped queue polling in `try/except redis.RedisError`, logged the failure, paused briefly, and retried instead of exiting. |
| `worker/worker.py` | 6-10 | The worker also ignored Redis authentication, so it could not consume jobs from a password-protected Redis container. | Added optional `REDIS_PASSWORD` support to the worker Redis client so it can run against secured Redis in Compose. |
| `frontend/app.js` | 6 | The frontend proxy was hardcoded to `http://localhost:8000`, which fails inside containers because the frontend container must call the API service over the container network. | Replaced the literal API URL with environment-driven service discovery so the frontend can build the upstream API URL from `API_URL` or from `API_SERVICE_NAME` plus `API_PORT`. |
| `frontend/app.js` | 13, 22 | The frontend's upstream API calls had no configurable timeout, so a dead or unreachable API could leave browser requests hanging instead of failing fast. | Added a shared request helper that uses `API_TIMEOUT` via `AbortSignal.timeout` for both submit and status requests. |
| `frontend/app.js` | 16, 25 | The frontend masked all upstream failures as a generic 500 `"something went wrong"`, which hid useful API errors like 404 job-not-found responses. | Added explicit upstream error handling and preserved the upstream HTTP status for the status route when available. |
| `frontend/app.js` | 29-30 | The listen port had been fixed earlier to `5000`, but it was still hardcoded and could not be changed by deployment configuration. | Switched the listener to `FRONTEND_PORT` from the environment while preserving `5000` as the default. |
| `frontend/app.js` | 1-17 | The Node frontend used Axios, which brings in `follow-redirects`. Under repeated proxy requests in this runtime, that path was emitting `MaxListenersExceededWarning` on sockets because extra `error` listeners accumulated on reused connections. | Removed Axios entirely and replaced the proxy calls with native Node `fetch` plus `AbortSignal.timeout`, preserving timeout handling and upstream status propagation without the listener warning path. |
| `frontend/app.js` | n/a | The frontend had no dedicated health endpoint, so a container health check would have to depend on static file behavior instead of a clear service-level signal. | Added a lightweight `/health` route so the container can report readiness without relying on the browser-facing UI route. |
| `frontend/package.json` | 8-10 | `axios` became an unnecessary runtime dependency after the proxy logic was moved to native `fetch`, leaving the manifest out of sync with the actual application code. | Removed `axios` from the frontend dependency list so the manifest matches the code that now runs in Node 22. |
| `frontend/package-lock.json` | 10-13 | The lockfile still described an Axios-based install, so a fresh install would keep pulling in the dependency tree that triggered the listener warning. | Regenerated the lockfile after uninstalling Axios so installs match the new native `fetch` implementation. |
| `frontend/views/index.html` | 23-28 | Submitting a job assumed every response was successful. On an error response, the page would try to read `job_id` anyway and display misleading or broken output. | Added `res.ok` checks, surfaced the backend error message to the user, and only started polling after a successful submission. |
| `frontend/views/index.html` | 31-37 | Polling also assumed success. If the backend returned an error or 404, the page kept rendering undefined state or could continue polling bad data. | Added response validation in `pollJob`, stopped treating failed responses as valid job states, and rendered the error on the affected job entry. |
| `frontend/views/index.html` | 48 | The rendered separator contained mojibake (`Ã¢â‚¬â€`) instead of a clean readable separator, which is a text-encoding bug in the UI. | Replaced it with a plain ASCII `-` so the job status text renders consistently across environments. |
| `api/Dockerfile` | 1-4 | The original Dockerfile stopped at dependency installation, had no runtime stage, no startup command, no health check, and no non-root execution. It would not produce a production-ready API container. | Replaced it with a multi-stage Python image that installs dependencies into a virtualenv, copies only runtime assets, runs as a non-root user, and includes a working API health check. |
| `worker/Dockerfile` | n/a | The worker had no Dockerfile, so the service could not be containerized or health-checked at all. | Added a multi-stage Python Dockerfile that runs the worker as a non-root user and verifies Redis connectivity through a container health check. |
| `frontend/Dockerfile` | n/a | The frontend had no Dockerfile, so there was no production container image, health check, or non-root runtime for the web service. | Added a multi-stage Node Dockerfile that installs only production dependencies, runs as the built-in non-root `node` user, and health-checks the local frontend endpoint. |
| `api/.dockerignore` | n/a | The API build context had no ignore rules, so a broad copy could accidentally include `.env`, virtualenv files, or Python cache artifacts in a build context. | Added a service-specific `.dockerignore` that excludes secrets and local build debris from the API image context. |
| `worker/.dockerignore` | n/a | The worker build context had no ignore rules, which risked sending local virtualenv files and caches to Docker during builds. | Added a worker `.dockerignore` that strips local-only artifacts from the image context. |
| `frontend/.dockerignore` | n/a | The frontend build context had no ignore rules, so local `node_modules` could leak into builds and create inconsistent dependency state. | Added a frontend `.dockerignore` that excludes local dependency directories and npm debug output. |
| `docker-compose.yml` | n/a | The project had no stack definition to wire the three services together, gate startup on health, isolate Redis from the host, or apply resource limits. | Added a Compose stack that uses a named internal network, keeps Redis off the host, waits for healthy dependencies, sources configuration from environment variables, and sets CPU/memory limits for every service. |
| `.env.example` | n/a | The example environment file was empty, so there was no documented set of variables for building and running the stack safely. | Added a complete non-secret example environment file covering service ports, Redis credentials, health settings, network naming, and resource limits. |
| `.flake8` | n/a | The repository had no Python lint configuration, so a flake8 stage would be inconsistent and could drift depending on defaults. | Added a shared flake8 configuration with an explicit line-length limit and local artifact exclusions. |
| `eslint.config.js` | n/a | The repository had no ESLint configuration, so the JavaScript lint stage could not run in CI in a predictable way. | Added a flat ESLint config for the Node-based frontend server with explicit globals and basic correctness rules. |
| `api/tests/test_main.py` | n/a | There were no API unit tests, so the pipeline could not validate core job creation, job lookup, or health behavior before building containers. | Added four pytest tests for the API using a mocked Redis client, covering queueing, successful status lookup, missing jobs, and health-check failure behavior. |
| `api/requirements-dev.txt` | n/a | The project had no declared Python test or lint dependencies for CI, which would make the test stage rely on ad hoc installs. | Added a development requirements file for pytest, coverage, httpx, and flake8 so the API test job can install a stable toolchain. |
| `scripts/integration-test.sh` | n/a | The project had no scripted end-to-end check, so CI could not prove that the composed stack processed a job successfully or tear down cleanly on failure. | Added an integration test script that starts the full stack, waits for container health, submits a job through the frontend, polls until completion, and always tears the stack down in a shell trap. |
| `scripts/rolling-deploy.sh` | n/a | There was no reusable rolling deployment primitive to keep the old container alive while a candidate image proved its health. | Added a generic rolling deploy script that starts a candidate container, waits up to 60 seconds for health, aborts safely if it fails, and only then replaces the live container. |
| `scripts/deploy.sh` | n/a | The repository had no deploy automation to orchestrate a health-gated rollout of the application services in order. | Added a deploy script that creates the internal network, ensures Redis is healthy, and then rolls out the API, worker, and frontend in sequence using the rolling deploy helper. |
| `.github/workflows/ci-cd.yml` | n/a | The repository had no CI/CD pipeline, so linting, tests, image builds, security scans, integration checks, and deploy logic were all missing. | Added a GitHub Actions workflow on `ubuntu-latest` that enforces the required stage order `lint -> test -> build -> security scan -> integration test -> deploy`, pushes images to a local registry service in the build job, uploads coverage and SARIF artifacts, and restricts deploys to pushes on `main`. |
